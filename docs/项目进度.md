## 1. 使用规则（给后续 AI）

### 1.1 执行规则
- 每次只执行一个最小步骤，不得跨步骤提前实现后续内容。
- 执行前先阅读与当前步骤相关的现有代码和配置。
- 若当前步骤依赖前一步未完成，先停止并说明阻塞原因，不得跳步。
- 若发现实现需要调整，以当前仓库代码为准，但不得偏离蓝图中的主链路目标。
- 默认只在 `QMcalculator` 内工作；计算逻辑必须集中在 `@qm/calc-core`，UI 不承载公式。

### 1.2 回写规则
- 开始执行某一步前，将该步骤状态从 `待完成` 改为 `进行中`。
- 步骤完成后，必须将状态改为 `已完成`。
- 步骤完成后，必须同步补充：
  - 完成日期
  - 改动文件
  - 验证结果
  - 备注（可选）
- 如果执行失败或被阻塞：
  - 将状态改为 `阻塞`
  - 在“备注”中写明阻塞原因
  - 不得伪造“已完成”

### 1.3 状态约定
- `待完成`：尚未开始
- `进行中`：当前正在执行
- `已完成`：代码已落地，并完成基本验证
- `阻塞`：存在前置条件缺失或环境问题，当前无法继续

### 1.4 回写示例

```md
- [x] A1 冻结输入输出协议
  - 状态：已完成
  - 完成日期：2026-07-02
  - 改动文件：`packages/calc-core/src/types.ts`
  - 验证结果：`CalculationInput` / `CalculationResult` 字段齐全
  - 备注：后续增强输入在阶段 B/D 补齐
```

## 2. 当前执行原则

- 先打通「选干员 -> 选技能 -> 配敌参 -> 看 DPS/解释」主链路，再扩规则覆盖，再做发布闭环。
- 所有计算统一走 `calculateSkillDps()`，结果必须包含 `summary/schedule/streams/formula/ruleTrace/warnings`。
- 未知或歧义语义必须通过 `warnings` 显式暴露，不允许静默猜测。
- 涉及规则、数据或核心逻辑的改动，优先跑 `npm run quality:gate`。
- 当前主线：阶段 E 已验收通过；后续可择优推进首技能续对齐、E1 残留次技能深化，或产品向增强（AOE/HPS 深度等）。
- 首技能旧口径对齐仍未全量完成（约 32%），不阻塞完工定义，但应在后续迭代继续回写 `结果对比.md`。
- 不提前做：多目标 AOE 完整建模、PVP 场景；治疗 maxHp 疗等深层语义可后续扩展。
