# QMcalculator 中长期项目规划与执行路线图

## 1. 文档说明

- 文档目的：为 `QMcalculator` 提供一份覆盖 9-12 个月的结构化中长期项目规划，统一产品目标、工程节奏、资源投入、风险治理与进度跟踪口径。
- 适用范围：`apps/web`、`packages/calc-core`、`scripts`、`docs`、`versions` 及其相关 CI/CD、数据同步、规则审计与发布流程。
- 更新原则：
  - 每个阶段启动前必须复核本文件中的准入条件、目标范围与依赖关系。
  - 每个阶段结束后必须更新里程碑状态、风险跟踪表与阶段复盘记录。
  - 重大范围变更需由项目负责人发起变更评审，并同步修订本文件。

## 2. 项目核心愿景与战略目标

### 2.1 核心愿景

将 `QMcalculator` 从“可用的单目标 DPS 计算工具”建设为“具备规则引擎、审计能力、版本治理与可解释输出的长期维护型计算平台”，持续为普通玩家、攻略作者与项目维护者提供可信、可追溯、可扩展的明日方舟计算能力。

### 2.2 战略目标

1. 在首个稳定版本内完成单目标 DPS MVP 闭环，具备可解释报告、告警体系和基础质量门禁。
2. 将旧项目 `ArknightCalculator` 的核心知识资产迁移为标准化规则协议、回归样例和审计脚本。
3. 建立围绕 `calc-core` 的长期演进机制，使 UI、数据、规则、计算、发布相互解耦。
4. 建立可回溯的版本管理、回滚机制、风险预警和进度度量体系，降低后续扩展成本。
5. 为第二阶段以后的治疗、召唤物、多目标、元素伤害等能力预留清晰的架构扩展位。

### 2.3 战略约束

- UI 不直接解析原始 blackboard。
- 所有计算统一进入 `calculateSkillDps()`。
- 遇到未知、歧义或部分覆盖语义时必须输出 `warnings`，不得静默猜测。
- 本规划优先保证计算正确性、可解释性与可维护性，不以 UI 花哨度为阶段通过条件。

## 3. 核心价值定位

### 3.1 面向用户的价值

- 普通玩家：快速获得稳定的干员技能 DPS、总伤与关键面板结果。
- 攻略作者：可查看公式链路、规则命中与 warning，支持对外说明与校验。
- 维护者：能通过规则 ID、回归样例、审计输出快速定位问题与补齐覆盖。

### 3.2 面向工程的价值

- 将旧项目中分散在 UI 与模型代码中的计算能力收敛到 `packages/calc-core`。
- 将隐式 custom 经验沉淀为协议化、可测试、可审计、可追踪的规则体系。
- 建立从数据同步、规则迁移、质量门禁、版本快照到发布回滚的完整闭环。

### 3.3 成功判定标准

| 维度 | 成功标准 | 目标值 |
|---|---|---|
| 正确性 | Top20 优先技能回归样例通过率 | `>= 95%` |
| 可解释性 | 结果中可输出 `summary/schedule/streams/ruleTrace/warnings` | `100% 覆盖 MVP 主流程` |
| 可维护性 | 新增单条规则可定位到唯一规则文件与规则 ID | `100%` |
| 发布稳定性 | 质量门禁通过后再发布 | `100%` |
| 风险可见性 | 高风险歧义场景均输出 warning code | `100%` |

## 4. 项目全周期总览

### 4.1 周期规划

- 规划总周期：约 40-48 周
- 阶段数量：5 个执行阶段
- 发布节奏：
  - 阶段 0-1：以周为单位推进内核与 MVP 稳定化
  - 阶段 2-3：以双周迭代推进规则迁移与 Beta 验证
  - 阶段 4：以月度版本推进平台化能力与对外稳定发布

### 4.2 阶段一览

| 阶段 | 时间周期 | 阶段目标 | 关键输出 | 前置依赖 |
|---|---|---|---|---|
| 阶段0：项目启动与基线冻结 | 第 1-2 周 | 冻结范围、协议、角色与度量口径 | 路线图、角色分工、协议基线、风险台账初版 | 无 |
| 阶段1：MVP 计算闭环 | 第 3-10 周 | 打通数据、规则、计算、前端最小闭环 | MVP 可用版、质量门禁 v1、最小回归集 | 阶段0 完成 |
| 阶段2：规则迁移与 Alpha 验证 | 第 11-20 周 | 迁移第一批高价值规则并建立对照回归 | Top20 首批样例、规则状态文档、Alpha 试用版 | 阶段1 完成 |
| 阶段3：Beta 稳定化与数据治理 | 第 21-32 周 | 完善审计、版本治理、发布回滚与 Beta 稳定性 | Beta 稳定版、差异报告、发布快照机制 | 阶段2 完成 |
| 阶段4：正式发布与平台化扩展准备 | 第 33-48 周 | 形成对外稳定版本并完成下一阶段扩展预研 | v1 正式版、平台化路线、扩展技术方案 | 阶段3 完成 |

## 7. 风险管控机制

### 7.1 风险分级

- `R1`：高风险，可能导致结果严重失真、发布失败或重大返工
- `R2`：中风险，可能导致阶段延期、覆盖不足或质量下降
- `R3`：低风险，对局部效率或体验有影响，但可通过常规节奏处理

### 7.2 风险跟踪表

| 风险ID | 阶段 | 风险描述 | 等级 | 触发信号 | 应对措施 | 责任角色 | 状态 |
|---|---|---|---|---|---|---|---|
| RK-01 | 阶段1-3 | 规则语义歧义导致结果失真 | R1 | `WARN_AMBIGUOUS_SEMANTIC` 激增或关键样例偏差异常 | 追加 custom、进入人工复核、暂缓发布 | 计算内核负责人 | 待跟踪 |
| RK-02 | 阶段1-3 | 数据更新导致索引或规则失配 | R1 | `sync:data` 失败、未映射 key 增加 | 回滚数据版本、补充审计与差异报告 | 数据治理负责人 | 待跟踪 |
| RK-03 | 阶段2-4 | 新旧结果差异过大影响可信度 | R2 | 对照回归超出容差阈值 | 建立偏差清单、按优先级修复、补充说明 | QA/测试负责人 | 待跟踪 |
| RK-04 | 阶段1-4 | 范围膨胀挤压核心目标 | R2 | 非目标需求频繁插入迭代 | 项目负责人冻结范围，延期到扩展阶段 | 项目负责人 | 待跟踪 |
| RK-05 | 阶段3-4 | 发布流程不稳定导致线上回归 | R1 | 质量门禁与线上结果不一致 | 强制快照、回滚演练、统一 CI 入口 | DevOps/发布负责人 | 待跟踪 |
| RK-06 | 阶段2-4 | 文档与规则状态不同步 | R2 | 规则已改动但无对应文档或状态更新 | 合入前检查文档更新、纳入周检项 | 文档/规则审计负责人 | 待跟踪 |

### 7.3 风险处理流程

```mermaid
flowchart LR
  detect[识别风险] --> assess[分级评估]
  assess --> owner[指定责任角色]
  owner --> action[制定缓解措施]
  action --> verify[验证效果]
  verify --> close[关闭或持续跟踪]
```

### 7.4 风险升级规则

- 任一 `R1` 风险未闭环，不得推进正式发布。
- 连续两个迭代出现同类高风险问题，必须召开专题复盘并调整流程。
- 任一阶段若关键里程碑延迟超过 20%，项目负责人必须提交重排计划。
